# emgpython_student 四人开发分工方案

## 目标

基于当前 `STM32 + ADS1292 + Python上位机` 工程，将 `S-EMG-main` 中的手势识别能力迁移到 `emgpython_student`，并采用 `4 人并行开发` 的方式推进。

总体原则：

1. 保留当前工程已有的 `STM32/ADS1292` 采集与协议解析能力
2. 新增手势数据采集、特征提取、模型训练、实时识别模块
3. 通过清晰接口降低 4 人之间的耦合

---

## 一、四人分工总览

建议拆分为以下 4 个角色：

1. `A：设备与数据接入`
2. `B：数据采集与数据集构建`
3. `C：特征提取与模型训练`
4. `D：实时识别与主程序集成`

可以简单理解为：

- `A` 管数据怎么进来
- `B` 管训练数据怎么采
- `C` 管模型怎么训出来
- `D` 管模型怎么实时跑起来

---

负责当前工程底层采集链路的稳定封装，给上层模块提供统一的 EMG 数据输入接口。

2. 封装统一数据输出接口
3. 支持读取 `raw` 和 `filtered` 两种数据
4. 支持获取“新增样本块”与“最新窗口数据”
5. 验证不同采样率下接口输出是否稳定

### 建议新增文件

- `emg_stream_adapter.py`

### 建议输出接口

```python

```

### 交付物

1. 稳定的数据适配模块
2. 明确的输入输出格式说明
3. 示例代码，证明上层可直接拿到 `N x 2` 的 EMG 数据

### 完成标准

1. 能在 `500Hz` 下稳定读出双通道数据
2. 输出统一为 `numpy.ndarray`

1. 原始数据和滤波数据切换不一致

## 三、B 组：数据采集与数据集构建

### 主要职责

负责将实时 EMG 流转换为可训练的数据集。

### 依赖关系

依赖 `A` 提供统一的 EMG 数据接口。

### 核心任务

1. 设计手势采集流程
2. 定义手势标签体系
3. 支持多轮采集
4. 将采集到的原始时序数据保存成数据集
5. 为后续训练模块提供标准数据格式

### 推荐采集方式

建议按“时间段”采集，而不是按“固定点数”采集。

例如：

1. 倒计时 1 秒
2. 用户保持某手势 2 到 3 秒
3. 自动截取该段 EMG 数据
4. 按类别保存

### 建议新增文件

- `gesture_session.py`
- `emg_dataset.py`

### 建议接口

```python
def collect_gesture_segment(
    ads_data,
    gesture_name: str,
    duration_sec: float,
    use_filtered: bool = True,
) -> np.ndarray:
    ...

def save_dataset(dataset, output_path: str):
    ...
```

### 推荐输出数据格式

```python
{
    "front": [segment1, segment2, ...],
    "back": [segment1, segment2, ...],
    "left": [segment1, segment2, ...],
    "right": [segment1, segment2, ...],
}
```

每个 `segment` 建议为：

```python
shape = (N, 2)
```

### 交付物

1. 可重复执行的手势采集脚本
2. 标准格式的数据集文件
3. 采集流程说明文档

### 完成标准

1. 至少支持 4 类手势采集
2. 每类支持多轮采集
3. 保存的数据可直接被训练模块读取

### 风险点

1. 标签录入不一致
2. 采集时间窗不稳定
3. 不同轮次动作强度波动大

## 四、C 组：特征提取与模型训练

### 主要职责

负责把 `S-EMG-main` 中的识别能力迁移成当前工程可用的训练模块。

### 依赖关系

依赖 `B` 产出的训练数据集。

### 核心任务

1. 实现标准滑窗
2. 迁移并重构特征提取函数
3. 适配 `2 通道 + 可配置采样率`
4. 完成随机森林训练与评估
5. 保存模型及元信息

### 建议新增文件

- `emg_window.py`
- `emg_features.py`
- `emg_train.py`
- `model_io.py`

### 第一阶段推荐特征集

先实现轻量且稳定的特征：

1. `RMS`
2. `MAV`
3. `WL`
4. `ZC`
5. `SSC`
6. `VAR`
7. `AR`
8. `MPF`

不要直接照搬原工程实现，必须做以下改造：

1. 采样率 `fs` 由外部传入

3. 特征长度根据通道数自动计算
4. 去掉粗暴的 `except: pass`

### 模型建议

第一版继续使用：

- `RandomForestClassifier`

### 模型保存建议

保存内容不要只有模型本体，还应包括：

1. 标签映射
2. 采样率
3. 窗口大小

### 交付物

1. 可独立运行的训练模块
2. 模型评估结果
3. 可加载的模型文件
4. 特征说明文档

### 完成标准

1. 能读取 B 组数据集
2. 能生成训练特征矩阵
3. 能输出测试集准确率
4. 能保存模型并供 D 组加载

### 风险点

1. 两通道模型精度低于预期
2. 原始采样率与特征计算不一致
3. 窗口长度设置不合理

## 七、推荐开发顺序

为了提高并行效率，建议按以下顺序推进：

1. `A` 先完成数据适配层
2. `B` 与 `C` 并行开发
3. `C` 先完成离线训练闭环
4. `D` 最后接入在线识别

更具体地说：

### 第 1 步

`A` 输出统一数据接口。

这是整个项目的前置依赖。

### 第 2 步

`B` 做采集数据集，`C` 先搭建滑窗和特征框架。

### 第 3 步

`C` 基于 `B` 的数据集训练第一版模型。

### 第 4 步

`D` 使用 `A` 的实时数据输入和 `C` 的模型进行在线推理集成。

## 八、建议的阶段目标

目标：

1. D 将模型接入实时主循环
2. 实时显示预测类别

### 第二阶段验收标准

1. 主界面可实时显示手势识别结果
2. 实时性可以接受
3. 不破坏现有波形显示能力

## 第三阶段：效果优化

目标：

1. 增加更多特征
2. 调整窗口参数
3. 对比 `raw` 和 `filtered`
4. 视情况尝试更强模型

1. `sample_rate`
2. `window_size`
3. `stride`

5. `use_filtered`
